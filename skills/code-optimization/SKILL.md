# 代码优化

## 何时使用

用户希望做结构化代码优化：先**自动或按用户指定**圈定要看的代码范围、理清逻辑，再列出**可选**优化点（含原因与预期收益），**经用户确认**后才改代码。

**不适用或简化**：用户只要一句话点评不要求结构化清单与后续落地；生成 commit message 请用 [`git-commit`](../git-commit/SKILL.md) 技能。

---

## 用户既定流程

1. **确定要优化的代码范围**（见下文「代码来源自动判断」；用户有明确要求时以用户为准），并梳理逻辑
2. 从代码优雅性、性能、可靠性、是否可用设计模式等方面考虑，优化代码。优化点要合理，不要为了优化而优化。从全局角度考虑，比如是否有可复用的逻辑、是否有可整合的逻辑等
3. 列出可优化项、原因、预计提升
4. 用户确认哪些需要优化
5. 编写代码

---

## 代码来源自动判断

在**用户未另行指定**范围、基线或对比方式时，按以下优先级选定本次要梳理的代码（**第 0 条始终最高**，可覆盖后面各条）：

| 优先级 | 条件 | 行动 |
| --- | --- | --- |
| **0** | 用户明确要求了文件/目录/模块/分支、对比基线（如某 tag/分支）、或「只看某次提交」等 | **严格按用户要求**圈定范围与 `git diff`/`git show` 等命令；不套用下面的自动规则覆盖用户意图 |
| **1** | `git diff --cached` 相对 `HEAD` **非空**（暂存区有已跟踪的改动） | **优先梳理暂存区**：以 `git diff --cached` 为主；需要上下文时结合 `git show :<path>` 或读工作区文件 |
| **2** | 暂存区无上述 diff，但工作区对已跟踪文件有修改（`git status` 显示 modified 等） | 梳理**未提交的工作区改动**：以 `git diff`（相对索引的未暂存改动）为主；若需「相对 `HEAD` 的全部未提交」一览，可用 `git diff HEAD` 辅助，但说明当前无暂存 diff、焦点在工作区 |
| **3** | 暂存与工作区对已跟踪文件均无有效改动，且用户也未指定范围 | **全项目/仓库视角**：通过 README、入口文件、包配置、主要目录结构等弄清项目是做什么的；再基于架构与热点路径提出合理优化点（仍须有据，不编造未读到的代码行为） |

**说明**：

- 仅有**未跟踪**新文件且无用户指定时：归入优先级 **3** 的全局梳理更合适，并在输出中说明「当前无已跟踪文件的 git diff，已结合目录与未跟踪文件可见部分分析」。
- **无 git 仓库**时：无法使用上述 diff 规则；若用户指定了路径则按路径只读分析，否则说明局限并尽量基于可见文件给出有限建议。

## 代理执行说明

### 步骤 1：确定范围、Git 与逻辑梳理

- 在仓库根目录执行 `git status`；若有 git，根据**「代码来源自动判断」**表选定本次主 diff/阅读范围（用户已指定则跳过自动表、直接用用户条件）。
- 按选定范围执行对应命令，例如：`git diff --cached`、`git diff`、`git diff HEAD`、`git diff <用户给定 base>...` 等；**不编造**未出现在 diff/status/已读文件中的改动。
- 若为优先级 **3**（全项目）：除 status 外，主动阅读项目说明与关键入口，概括职责与模块边界，再收敛到可优化的具体位置。
- 用简短文字梳理：本次关注代码的**意图**或项目**目标**、主要**数据流/调用关系**、与周边模块的**耦合点**（为后续「全局」视角服务）。

### 步骤 2：分析原则与维度

在落实用户第 2 条时，同时遵守：

- **维度**：代码优雅性；性能；可靠性；设计模式是否**真的**适用（避免生硬套模式）；全局上是否有**可复用**或可**合并**的重复逻辑。
- **克制**：每项建议须有**可说明的依据**（性能需热点或复杂度/IO 等线索；可靠性需错误路径、边界、并发等线索）。拒绝「为优化而优化」、无关大重构。

### 步骤 3：输出优化清单（在用户确认前必须给出）

对每条建议使用下表格式（可按优先级排序：正确性/安全/可维护性优先，其次性能）：

| 编号 | 可优化项 | 原因 | 预计提升 |
| --- | --- | --- | --- |

- **预计提升**须标明类型（如可读性、性能、健壮性、可维护性）；能定量则定量，否则用「小幅/中等/显著」并写**前提或风险**（若有行为变化）。

### 步骤 4：等待用户确认

- **在用户提供明确选择（编号或条目名）之前，不编写用于落地的生产代码**（确认前仅只读：跑 git、读文件、澄清范围/基线）。
- 若用户表示「全部」：视为全选；**再复述一遍**将实施的条目清单，经用户简单确认（如「对，就这样」）后进入步骤 5。

### 步骤 5：编写代码

- **仅实现**用户确认的条目；命名、类型、风格与项目现有代码一致；**不做**未选中项的顺手重构。
- 完成后逐条对应说明：改了什么、对应哪条建议；对行为可能变化的项，给出建议的验证方式（测试命令或手动场景）。
