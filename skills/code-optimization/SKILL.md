# 代码优化

## 何时使用

用户希望**对照当前本地改动**做结构化优化：先看全量 diff、理清逻辑，再列出**可选**优化点（含原因与预期收益），**经用户确认**后才改代码。

**不适用或简化**：无 git 仓库/无有效 diff、或用户只要一句话点评不要求改代码；生成 commit message 请用 [`git-commit`](../git-commit/SKILL.md) 技能。

---

## 用户既定流程

1. 使用 git 检查当前改动的所有代码，梳理逻辑
2. 从代码优雅性、性能、可靠性、是否可用设计模式等方面考虑，优化代码。优化点要合理，不要为了优化而优化。从全局角度考虑，比如是否有可复用的逻辑、是否有可整合的逻辑等
3. 列出可优化项、原因、预计提升
4. 用户确认哪些需要优化
5. 编写代码

---

## 代理执行说明

### 步骤 1：Git 与逻辑梳理

- 在仓库根目录执行 `git status`。
- **默认**用 `git diff HEAD` 查看相对**上一次提交**的全部已跟踪文件改动（工作区 + 暂存区相对 `HEAD` 的总览）。若用户指定对比基线（如某分支、某 tag），改用用户给出的 `git diff <base>...HEAD` 或等价命令。
- 无 diff 时：结合 `git status` 说明（例如无改动、仅有未跟踪文件）；**不编造**未出现在 diff/status 中的改动。
- 用简短文字梳理：本次改动**意图**、主要**数据流/调用关系**、与周边模块的**耦合点**（为后续「全局」视角服务）。

### 步骤 2：分析原则与维度

在落实用户第 2 条时，同时遵守：

- **维度**：代码优雅性；性能；可靠性；设计模式是否**真的**适用（避免生硬套模式）；全局上是否有**可复用**或可**合并**的重复逻辑。
- **克制**：每项建议须有**可说明的依据**（性能需热点或复杂度/IO 等线索；可靠性需错误路径、边界、并发等线索）。拒绝「为优化而优化」、无关大重构。

### 步骤 3：输出优化清单（在用户确认前必须给出）

对每条建议使用下表格式（可按优先级排序：正确性/安全/可维护性优先，其次性能）：

| 编号 | 可优化项 | 原因 | 预计提升 |
| --- | --- | --- | --- |

- **预计提升**须标明类型（如可读性、性能、健壮性、可维护性）；能定量则定量，否则用「小幅/中等/显著」并写**前提或风险**（若有行为变化）。

### 步骤 4：等待用户确认

- **在用户提供明确选择（编号或条目名）之前，不编写用于落地的生产代码**（确认前仅只读：跑 git、读文件、澄清范围/基线）。
- 若用户表示「全部」：视为全选；**再复述一遍**将实施的条目清单，经用户简单确认（如「对，就这样」）后进入步骤 5。

### 步骤 5：编写代码

- **仅实现**用户确认的条目；命名、类型、风格与项目现有代码一致；**不做**未选中项的顺手重构。
- 完成后逐条对应说明：改了什么、对应哪条建议；对行为可能变化的项，给出建议的验证方式（测试命令或手动场景）。
